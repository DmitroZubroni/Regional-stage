## 3) ==Отчёт по логике== 

| 1   | Неверная логика AccessControl, Разрешено всем, кроме роли,нужно исправить `==` вместо `!=`                                                                                                                                                                       |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2   | `addMinute()` доступна всем, любой меняет фазы. Нужно ограничить доступ (onlyOwner) или удалить данный метод вовсе                                                                                                                                               |

| 4   | Ошибка в `takeMyAllowance`. Баланс owner увеличивается (owner += amount ). Должен уменьшаться owner -= amount                                                                                                                                                    |
| 5   | `bf = bf;`                                                                                                                                                                                                                                                       |
| 6   | transferToken - неверная проверка if (msg.sender != owner) проверяет availableOwnerTokens. Однако эта переменная глобальная, а не привязана к конкретному пользователю. Это блокирует переводы для всех обычных пользователей, если владелец не "разрешил" объем |

## 5) ==Отчёт по информационной безопасности==

| 1)  | Пароли хранятся в` passwordMap` в виде хеша кессак. В блокчейне все данные публичны. Хеш можно легко перехватить в момент транзакции signUp, нужно отказаться от логинов и паролей и перейти на msg.sender |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2)  | reentrancy атака при `payable(owner).transfer(msg.value)`, тут нужно добавить call или ReentrancyGuard, что бы данные изменялись вовремя и проверку на наличие средств у пользователя                      |
| 3)  | DoS/DDos атаки через массивы `for (uint256 i = 0; i < requests.length; i++)` может стать слишком дорогим, нужно использовать mapping вместо массива                                                        |
| 4)  | Манипуляция временем: Функция `addMinute` публична. Любой пользователь может вызывать её бесконечно, искусственно "проматывая" время контракта, чтобы преждевременно завершить фазы                        |
| 5)  | Отсутствие проверок на переполнение/недостаток в transferToProvider. При передаче токенов провайдерам нет проверки, что у owner достаточно токенов для передачи (privAmount, pubAmount)                    |

## 6) ==Оптимизация смарт-контракта==

| 1   | Whitelist как структура, Использовать `mapping(address => bool)`                                                                                                     |
| --- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2   | Избыточная структура User, убрать все поля кроме role                                                                                                                |
| 3   | Использование `uint256`, заменить на `uint8/16/32/...`                                                                                                               |
| 4   | Массив Whitelist,  удалить, заменить на mapping                                                                                                                      |
| 5   | mapping loginMap, passwordMap Удалить                                                                                                                                |
| 6   | Логика фаз токенов реализована в смарт контракте, а нужно сделать отдельные смарт контракты токенов для каждой фазы                                                  |
| 7   | Использовать `transferFrom()` `aprove` и удалить  массив approveList                                                                                                 |
| 8   | использовать memory вместо storage                                                                                                                                   |
| 9   | не нужные переменные, избавиться от Time_dif                                                                                                                         |
| 10  | Заменить массив requests и цикл в sendRequestToWhitelist на mapping(address => bool) hasRequested. Это сократит стоимость проверки                                   |
| 11  | Переменные privPhase, seedPhase, dec не меняются. Их следует объявить как constant                                                                                   |
| 12  | в реализации данной бизнес логики в смарт контракте не нужна функция авторизации от неё можно избавиться, а авторизацию реализовать на фронтенд части через метамас  |
